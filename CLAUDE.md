## Projekt
**Atelier** = kuratierter Premium-Marktplatz für fertige Software (SaaS, Apps, ganze Firmen, Anteile).
Enterprise-Ziel: HA, mandantenfähig, in **~4 Wochen** marktreif für echte **zahlende** Kunden — Produktionsqualität, keine Demo.
> **PRIORITÄT — Owner-Direktive 2026-06-04 (verbindlich):** Atelier wird **von Anfang an produktionsreif für den weltweiten Markt** gebaut — nicht „erst DE, später Welt", keine Wegwerf-Zwischenstufen, kein „später härten". Jede Entscheidung (Architektur, Zahlung, Recht, Daten, UI) trifft ab Tag 1 den höchsten Produktionsstandard für globale Nutzer. Ab sofort mitgedacht: **i18n** (Mehrsprachigkeit statt hartkodiertem Deutsch), **Mehrwährung + internationale Zahlung/Steuer** (Stripe), **DSGVO + internationale Rechts-/Compliance-Texte**, weltweite Performance/Erreichbarkeit. Tempo opfert nie Produktionsreife. **Ehrlicher Trade-off:** „weltweit ab Start" vergrößert den Scope (v. a. i18n + Mehr-Jurisdiktions-Recht/Steuer) → kann die ~4-Wochen-Marke dehnen; wird bei jeder Priorisierung offen benannt statt still gekürzt.
**Vollständiger Plan + Architektur-Entscheidungen: `docs/ROADMAP.md`** (Zahlungen/Stripe, eigene Domain+Resend, Trust Center/Recht, „Individuell"-Tarif, isoliertes **Staff Center** + **Support Center**). Owner-Fragen dort beantwortet: Multi-Standort/SSO/Hetzner = vorerst **nicht** nötig; Multi-Org/RLS bleibt Pflicht.
Stack: **Vite + React 18 + TS** · **Supabase Cloud** (Postgres+Auth+Storage+RLS) · **Cloudflare Pages** (Hosting) · Docker/Traefik als Self-Host-Schalter.
Design ist **GESPERRT**, 1:1 aus `atelier.html`.

## Produktvision & Scope — Owner-Projektbeschreibung (`11_Atelier/projektbeschriebung.md`, 2026-06)

## Enterprise-Regeln
- **Skalierbar by default, schlank deployen, am 1. Kunden umlegen.** Teure Infra liegt als Code bereit (`infra/`), läuft aber nicht idle. Siehe `docs/SCALING.md`.
- **Mandantenfähig ab Migration #1:** `org_id` auf jeder Business-Zeile, **RLS** erzwingt Isolation. Jede neue Tabelle: `enable row level security` + Policies in derselben Migration.
- **React ≠ Sicherheit:** die DB (RLS) sichert ALLES ab. anon-Key ist öffentlich (sicher via RLS); `service_role`-Key NIE im Client/Repo.
- **Keine Secrets in Git:** `.env` (gitignored), nur `.env.example` getrackt.
- **Kein Fake-Data in echten Flows:** statische `src/data/*` ist MVP-Platzhalter; sobald Backend live, aus Supabase laden (`src/lib/supabase.ts`).
- **Doku ab Tag 1 Pflicht:** README, ARCHITECTURE, SCALING, RUNBOOK, SECURITY, ADRs. Architekturentscheidungen als ADR.
